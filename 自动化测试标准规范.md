# 自动化测试标准规范

版本：V1.0  
生效日期：2026-05-16  
适用范围：功能测试、接口自动化测试、Web UI 自动化测试、App 自动化测试、回归测试、冒烟测试。

## 1. 目标

本规范用于统一测试用例设计、自动化脚本开发、测试执行、结果报告和维护标准，确保自动化测试具备以下能力：

- 需求、测试用例、自动化脚本、测试报告可追溯。
- 核心业务流程、异常场景、边界条件覆盖完整。
- 自动化脚本稳定、可重复执行、可维护。
- 测试失败后能够快速定位问题原因。
- 自动化测试结果可作为版本发布质量评估依据。

## 2. 术语定义

| 术语 | 说明 |
|---|---|
| 测试用例 | 针对某个需求或业务规则设计的测试验证项 |
| 自动化脚本 | 将测试用例转化为可自动执行的代码 |
| 冒烟测试 | 验证系统核心功能是否可用的最小测试集 |
| 回归测试 | 验证变更未破坏已有功能的测试集 |
| 边界测试 | 针对输入范围、长度、数量、时间等边界值进行验证 |
| 断言 | 自动化脚本中用于判断实际结果是否符合预期的校验逻辑 |
| 覆盖矩阵 | 需求、用例、脚本、执行结果之间的映射关系 |

## 3. 测试用例设计规范

### 3.1 用例必填字段

每条测试用例必须包含以下字段：

| 字段 | 说明 | 是否必填 |
|---|---|---|
| 用例编号 | 唯一标识，例如 `TC-LOGIN-001` | 是 |
| 需求编号 | 对应需求、缺陷或用户故事编号 | 是 |
| 模块名称 | 所属系统模块 | 是 |
| 用例标题 | 简要描述验证目标 | 是 |
| 前置条件 | 执行前必须满足的环境、账号、数据条件 | 是 |
| 测试数据 | 输入参数、账号、文件、业务数据等 | 是 |
| 操作步骤 | 明确、可执行、无歧义的操作步骤 | 是 |
| 预期结果 | 明确的页面、接口、数据库或业务状态结果 | 是 |
| 优先级 | `P0`、`P1`、`P2`、`P3` | 是 |
| 用例类型 | 正常、异常、边界、权限、兼容、安全等 | 是 |
| 是否自动化 | 是、否、待评估 | 是 |
| 自动化脚本路径 | 自动化后对应脚本路径 | 自动化时必填 |
| 维护状态 | 有效、废弃、待更新 | 是 |

### 3.2 用例编号规则

推荐格式：

```text
TC-模块-类型-序号
```

示例：

```text
TC-LOGIN-NORMAL-001
TC-LOGIN-BOUNDARY-002
TC-ORDER-EXCEPTION-003
TC-PAYMENT-API-004
```

常用类型标识：

| 类型 | 标识 |
|---|---|
| 正常流程 | `NORMAL` |
| 异常流程 | `EXCEPTION` |
| 边界测试 | `BOUNDARY` |
| 接口测试 | `API` |
| UI 测试 | `UI` |
| 权限测试 | `AUTH` |
| 兼容性测试 | `COMPAT` |
| 安全测试 | `SECURITY` |

### 3.3 用例优先级定义

| 优先级 | 定义 | 自动化建议 |
|---|---|---|
| P0 | 核心主流程、阻塞发布的关键功能 | 必须自动化 |
| P1 | 高频回归、重要业务规则、重要异常场景 | 优先自动化 |
| P2 | 一般功能、常规边界、低频业务场景 | 视成本自动化 |
| P3 | 低频场景、一次性验证、人工判断强的场景 | 通常不自动化 |

## 4. 测试覆盖标准

### 4.1 基础覆盖要求

每个需求至少应覆盖以下内容：

- 正常业务流程。
- 异常业务流程。
- 输入为空、缺失、格式错误。
- 数值、长度、数量、时间等边界条件。
- 权限、角色、状态流转限制。
- 重复提交、幂等、并发或快速连续操作。
- 数据保存、修改、删除后的状态校验。
- 错误提示、错误码、日志记录。

### 4.2 边界测试覆盖要求

对于存在范围限制的字段，必须覆盖：

| 边界类型 | 示例 |
|---|---|
| 最小值以下 | `min - 1` |
| 最小值 | `min` |
| 最小值以上 | `min + 1` |
| 中间有效值 | 正常范围内典型值 |
| 最大值以下 | `max - 1` |
| 最大值 | `max` |
| 最大值以上 | `max + 1` |
| 空值 | `null`、空字符串、未传字段 |
| 非法类型 | 字母、特殊符号、小数、对象、数组等 |

示例：年龄字段规则为 `18 <= age <= 60`。

| 测试值 | 预期结果 | 用例类型 |
|---|---|---|
| 17 | 不通过 | 边界外 |
| 18 | 通过 | 最小边界 |
| 19 | 通过 | 边界内 |
| 30 | 通过 | 正常值 |
| 59 | 通过 | 边界内 |
| 60 | 通过 | 最大边界 |
| 61 | 不通过 | 边界外 |
| 空值 | 不通过 | 异常值 |
| 字符串 | 不通过 | 类型错误 |

### 4.3 常见场景覆盖清单

| 场景 | 必测点 |
|---|---|
| 数值范围 | 最小值、最大值、边界内、边界外、负数、零、小数 |
| 字符长度 | 空字符串、1 个字符、最小长度、最大长度、超过最大长度 |
| 金额 | 0、负数、最小金额、最大金额、小数精度、超大金额 |
| 日期时间 | 开始时间、结束时间、跨天、跨月、跨年、闰年、过期时间 |
| 枚举值 | 全部合法枚举、非法枚举、空值、大小写差异 |
| 文件上传 | 空文件、最小文件、最大文件、超大文件、非法格式 |
| 分页查询 | 首页、尾页、空结果、页码为 0、页大小超限 |
| 权限控制 | 无权限、只读权限、编辑权限、管理员权限 |
| 状态流转 | 合法流转、非法流转、重复操作、撤回、取消 |

## 5. 自动化准入标准

### 5.1 适合自动化的用例

满足以下条件的用例优先自动化：

- 冒烟测试用例。
- P0 核心主流程用例。
- 高频回归用例。
- 接口稳定、数据可控的用例。
- 预期结果明确、可通过断言判断的用例。
- 手工执行成本高、重复频率高的用例。

### 5.2 不建议自动化的用例

以下用例不建议直接自动化：

- 只执行一次的临时验证。
- 需求、页面或接口频繁变动且尚未稳定。
- 强依赖人工视觉判断的场景。
- 第三方环境不稳定且无法模拟的场景。
- 执行成本高于维护收益的低频用例。

## 6. 自动化脚本开发规范

### 6.1 脚本命名

推荐格式：

```text
test_模块_功能_场景
```

示例：

```text
test_login_success.py
test_login_password_error.py
test_user_age_boundary.py
test_order_create_api.py
```

### 6.2 目录结构

推荐结构：

```text
automation-tests/
  cases/
    api/
    ui/
  data/
  fixtures/
  pages/
  clients/
  utils/
  reports/
  logs/
  config/
```

目录说明：

| 目录 | 说明 |
|---|---|
| `cases` | 测试脚本 |
| `data` | 测试数据 |
| `fixtures` | 前置和后置处理 |
| `pages` | UI 页面对象 |
| `clients` | 接口请求封装 |
| `utils` | 公共工具方法 |
| `reports` | 测试报告 |
| `logs` | 执行日志 |
| `config` | 环境配置 |

### 6.3 脚本基本结构

自动化脚本应包含：

- 测试数据准备。
- 前置条件处理。
- 执行测试步骤。
- 明确断言。
- 日志记录。
- 测试数据清理。
- 异常截图或响应记录。

示例：

```python
def test_login_success():
    username = "test_user"
    password = "Password123"

    response = login(username, password)

    assert response.status_code == 200
    assert response.json()["code"] == "SUCCESS"
    assert response.json()["data"]["username"] == username
```

### 6.4 编码要求

- 测试脚本必须可独立运行，不依赖其他用例执行结果。
- 公共逻辑必须封装复用，禁止大量重复代码。
- 测试数据不得硬编码在复杂业务逻辑中。
- 禁止使用固定等待替代稳定等待机制。
- 禁止吞掉异常后继续标记用例通过。
- 禁止只打印结果、不做断言。
- 日志中不得输出真实密码、密钥、Token、身份证号等敏感信息。

## 7. 断言规范

### 7.1 通用要求

每条自动化用例必须包含有效断言。禁止仅判断“程序未报错”或“接口有返回”。

断言应覆盖：

- 状态码。
- 业务码。
- 关键字段。
- 数据类型。
- 数据内容。
- 状态变化。
- 错误信息。

### 7.2 接口测试断言

接口测试至少断言：

- HTTP 状态码。
- 业务状态码。
- 响应字段是否完整。
- 响应字段类型是否正确。
- 核心字段值是否符合预期。
- 异常场景下错误码和错误信息是否正确。
- 必要时校验数据库或后续查询结果。

### 7.3 UI 测试断言

UI 测试至少断言：

- 页面跳转是否正确。
- 关键元素是否展示。
- 按钮、输入框、菜单状态是否正确。
- 提示信息是否正确。
- 表格、列表、详情页数据是否正确。
- 保存、删除、提交后业务状态是否正确。

## 8. 测试数据规范

### 8.1 数据要求

- 测试数据应与测试逻辑分离。
- 自动化测试不得使用生产真实敏感数据。
- 测试数据应可重复创建和清理。
- 用例执行前应准备所需数据，执行后应清理脏数据。
- 边界值、异常值、枚举值应集中维护。

### 8.2 数据文件示例

```json
[
  {
    "case_id": "TC-USER-BOUNDARY-001",
    "age": 17,
    "expected_code": "INVALID_AGE"
  },
  {
    "case_id": "TC-USER-BOUNDARY-002",
    "age": 18,
    "expected_code": "SUCCESS"
  }
]
```

## 9. 环境与配置规范

自动化测试环境应满足：

- 环境配置与脚本分离。
- 不同环境使用独立配置文件，例如开发、测试、预发。
- 账号、密码、Token、密钥通过环境变量或密钥管理工具维护。
- 测试环境数据应可控、可恢复。
- 外部依赖应具备 Mock、Stub 或稳定测试环境。
- 执行前检查服务可用性、数据库连接、浏览器驱动或依赖版本。

## 10. 执行规范

### 10.1 执行分类

| 类型 | 执行时机 | 通过标准 |
|---|---|---|
| 冒烟测试 | 每次提测、部署后 | P0 用例 100% 通过 |
| 回归测试 | 版本发布前 | P0 100% 通过，P1 通过率不低于 95% |
| 接口自动化 | 每日构建、接口变更后 | 失败必须分析 |
| UI 自动化 | 每日构建、发布前 | 失败必须截图和记录日志 |
| 定时巡检 | 每日或每周固定时间 | 核心链路可用 |

### 10.2 失败处理

测试失败后必须区分：

- 产品缺陷。
- 脚本问题。
- 测试数据问题。
- 环境问题。
- 第三方依赖问题。

失败用例必须记录：

- 用例编号。
- 执行环境。
- 输入数据。
- 期望结果。
- 实际结果。
- 错误日志。
- 截图或接口响应。
- 初步原因分类。

## 11. 报告规范

自动化测试报告至少包含：

- 执行时间。
- 执行环境。
- 执行人或触发方式。
- 代码分支或版本号。
- 用例总数。
- 通过数。
- 失败数。
- 跳过数。
- 通过率。
- 失败用例明细。
- 错误日志、截图、接口响应。
- 历史趋势，适用于持续集成场景。

质量门禁建议：

| 指标 | 标准 |
|---|---|
| 冒烟测试通过率 | 100% |
| P0 用例通过率 | 100% |
| P1 用例通过率 | 不低于 95% |
| 自动化脚本误报率 | 不高于 5% |
| 失败用例分析率 | 100% |

## 12. 覆盖矩阵规范

每个自动化脚本必须能追踪到测试用例和需求。

| 需求编号 | 用例编号 | 用例标题 | 用例类型 | 自动化脚本 | 执行状态 |
|---|---|---|---|---|---|
| REQ-001 | TC-LOGIN-001 | 正确账号密码登录 | 正常流程 | `cases/api/test_login_success.py` | 已自动化 |
| REQ-001 | TC-LOGIN-002 | 密码错误登录失败 | 异常流程 | `cases/api/test_login_password_error.py` | 已自动化 |
| REQ-002 | TC-USER-001 | 年龄边界校验 | 边界测试 | `cases/api/test_user_age_boundary.py` | 已自动化 |

覆盖矩阵必须满足：

- 每个需求至少有一条测试用例覆盖。
- 每条 P0、P1 用例必须明确是否自动化。
- 自动化脚本路径必须可定位。
- 废弃用例必须标记状态，不得直接删除历史记录。

## 13. 评审规范

### 13.1 测试用例评审

测试用例评审应检查：

- 是否覆盖所有需求点。
- 是否覆盖正常、异常、边界、权限场景。
- 预期结果是否明确。
- 测试数据是否可准备。
- 是否存在重复或无效用例。
- 是否标明自动化优先级。

### 13.2 自动化脚本评审

自动化脚本评审应检查：

- 脚本是否对应明确用例编号。
- 是否具备有效断言。
- 是否可独立运行。
- 是否具备数据准备和清理能力。
- 日志和失败信息是否可定位。
- 是否存在固定等待、硬编码、敏感信息泄露。
- 是否符合目录和命名规范。

## 14. 维护规范

自动化测试资产应持续维护：

- 需求变更后，同步更新测试用例和自动化脚本。
- 接口、页面、字段变更后，及时更新断言和测试数据。
- 失效用例应标记为废弃或待更新，不得无记录删除。
- 连续失败的自动化脚本必须分析并修复。
- 不稳定脚本不得纳入发布质量门禁。
- 每个版本发布后复盘自动化覆盖率、失败率和误报率。

## 15. 完成标准

一条测试用例完成自动化，必须同时满足：

- 测试用例已评审通过。
- 自动化脚本已开发完成。
- 脚本可独立运行。
- 脚本包含明确断言。
- 测试数据可准备、可清理。
- 执行结果进入测试报告。
- 用例编号、需求编号、脚本路径已建立追踪关系。
- 失败时能够输出可定位的日志、截图或接口响应。

最终标准：

```text
需求有用例覆盖，用例有脚本覆盖，脚本有断言验证，执行有报告证明，失败可快速定位。
```
